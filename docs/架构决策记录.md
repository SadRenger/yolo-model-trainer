## ADR-1: 选择 Tauri 2.x 而非 Electron

**状态：** ✅ 已采纳 | 2026-07-17

### 背景

需要一款 Windows 桌面应用框架。前端使用 Web 技术，后端需管理 Python 子进程和文件系统。两个候选：Electron 和 Tauri。

### 决策

**选择 Tauri 2.x。**

### 理由

| 维度 | Tauri 2.x | Electron | 评估 |
|---|---|---|---|
| 安装包基础体积 | ~5 MB | ~150 MB+ | 本项目的体积大头是 PyTorch（~2GB），框架体积应尽量小 |
| 内存占用 | 低（系统 WebView2 共享） | 高（独立 Chromium） | 训练任务本身大量消耗 GPU/内存 |
| 后端语言 | Rust | Node.js | Rust 的进程管理更可靠，资源清理更可控 |
| Python 子进程 | `std::process::Command` 原生支持 | `child_process` 可用但无所有权保证 | Rust 编译期保证进程句柄不易泄漏 |
| 前端运行时 | 系统 WebView2（Win10 1803+ 已内置） | 内嵌 Chromium | 无需额外打包 |

### 后果

**正面：** 安装包除去 Python 依赖后体积极小；训练中前端内存占用可忽略；Rust 编译期保证降低进程泄漏风险。

**负面：** 前端调试依赖 WebView2 DevTools；Rust 学习曲线较陡（但 AI 写 Rust 无压力）；部分 CSS 在 WebView2 中可能与 Chrome 不一致。

## ADR-2: 前端使用原生 JavaScript（无框架）

**状态：** ✅ 已采纳 | 2026-07-17

### 背景

4 个主页面 + ~10 个全局组件。候选：React/Vue 或原生 JS。

### 决策

**选择原生 JavaScript，零框架 SPA。**

### 理由

| 维度 | 原生 JS | React / Vue |
|---|---|---|
| 项目规模匹配 | 4 页面不需要框架级状态管理 | 适合任意规模但过重 |
| 构建步骤 | 零（浏览器直接运行） | 需要 Vite/Webpack |
| AI 开发效率 | AI 对原生 DOM API 训练数据最丰富 | 需理解 JSX/SFC 模板和框架生命周期 |
| 上下文透明度 | 代码即最终形态，WYSIWYG | 虚拟 DOM、响应式系统等隐式行为 |
| 长期维护 | 本项目一次性交付（需求 1.5），不设后续迭代 | 框架利于长期维护但本次用不上 |

### 后果

**正面：** 零构建步骤；AI 开发效率最高；调试直观无 source map 映射。

**负面：** 缺少框架的响应式状态管理（手动实现 `state.js`）；组件复用靠手动封装；未来若扩展为大型应用需重构。

## ADR-3: Python 运行时嵌入安装包

**状态：** ✅ 已采纳 | 2026-07-17

### 背景

核心计算依赖 PyTorch + Ultralytics。候选：嵌入式 Python、要求用户自行安装、Sidecar 服务模式。

### 决策

**选择方案 A：嵌入式 Python 3.13，打包进安装包。**

### 理由

| 维度 | 嵌入式 Python | 系统 Python | Sidecar 服务 |
|---|---|---|---|
| 用户体验 | ⭐⭐⭐⭐⭐ 双击即用 | ⭐ 需手动配置环境 | ⭐⭐ 需额外启动服务 |
| 版本一致性 | 锁定版本 | 用户版本各异 | 可锁定但需额外部署 |
| 安装包体积 | ~2.0-2.4 GB | 不占空间 | 仅含运行时 |
| 隔离性 | 完全独立 | 可能与系统 Python 冲突 | 进程隔离 |
| 需求匹配 | 完美匹配"不懂代码的用户" | 与核心定位矛盾 | 多一层运维复杂度 |

**关键考量：** 产品的核心定位是"让不懂代码的人也能训练 YOLO 模型"。2.0-2.4 GB 的安装包体积是"全离线 GPU 训练环境"的必要代价，也是产品核心卖点。

### 后果

**正面：** 用户无需技术知识；版本锁定保证行为一致；不污染系统 Python。

**负面：** 安装包体积大；NSIS 打包复杂；更新 PyTorch/Ultralytics 需重新分发。

## ADR-4: 训练日志通道与报告数据通道分离

**状态：** ✅ 已采纳 | 2026-07-17

### 背景

训练中同时需要实时日志展示和完整训练报告。最初考虑共用一个数据通道。

### 决策

**将"日志通道"和"报告数据通道"完全分离。**

```
训练进程 stdout       → Rust 管道 → 前端日志终端（受"日志级别"控制展示量）
训练进程 metrics 文件  → Rust 读取 → HTML 报告生成（始终完整，不受日志级别影响）
```

### 理由

| 维度 | 双通道分离 | 单一通道 |
|---|---|---|
| 日志级别灵活性 | 切换级别不影响报告 | 降低级别 → 报告数据也丢失 |
| 数据完整性 | metrics 文件 100% 数据保真 | stdout 可能被 5000 行限制截断 |
| 前端性能 | 日志终端只渲染文本 | 每行都解析 JSON，数据量大时卡顿 |
| 报告可复现 | 随时从文件重新生成 | 强依赖 stdout 流，丢失不可恢复 |

**关键洞察：** 日志是实时展示流（可丢弃/截断/过滤），报告是静态生成产物（必须完整、可复现）。本质不同的两件事，不应共享数据源。

### 后果

**正面：** 日志级别不影响报告完整性；报告可从 metrics 随时重生成。

**负面：** Python 需维护双通道输出；Rust 需同时监听管道和文件；两通道数据可能短暂不一致。

## ADR-5: 安装包不内置模型文件

**状态：** ✅ 已采纳 | 2026-07-17

### 背景

YOLO 训练需要预训练权重文件（.pt）。候选：打包进安装包，或用户自行下载。

### 决策

**不内置模型文件。用户在设置页自行下载。**

### 理由

| 维度 | 不内置模型 | 内置模型 |
|---|---|---|
| 安装包体积 | ~2.0-2.4 GB（仅 PyTorch） | +50-500 MB |
| 法律/许可证 | 无需处理再分发许可 | 需逐一确认 |
| 版本灵活性 | 用户可选任意 YOLO 版本 | 内置什么就用什么 |
| 用户认知 | "你需要选择适合自己的模型" | 误解"内置即最好" |
| 维护成本 | 只维护下载链接 | 每次发新模型都要更新安装包 |

### 后果

**正面：** 安装包不膨胀；用户可选任意版本和尺寸。

**负面：** 首次使用必须联网下载——这是唯一额外步骤；下载链接需随 Ultralytics 发版同步更新。
