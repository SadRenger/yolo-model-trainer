## 一、设计原则

### 1.1 为什么需要状态码

- **你反馈快**：给我一个码（比如 `T-104`），我 30 秒内定位到具体文件和行号
- **我定位快**：不需要追问"训练跑到第几轮了？你用的什么参数？"
- **诊断面板可读**：IPC 面板每行都带码，每层间的消息跳转一目了然

### 1.2 编码规则

```
{模块前缀}-{序号}{类型后缀}
```

- **模块前缀**：一个字母，标识功能模块
- **序号**：三位数字，同模块内唯一
- **类型后缀**：无后缀 = 正常状态，`E` = 错误，`W` = 警告

示例：`T-003`（训练启动）→ `T-104`（进度更新）→ `T-104E`（GPU 显存溢出）

### 1.3 序列设计

每个模块的状态码**从小到大按执行顺序排列**，所以看相邻两条的码号就能判断执行到哪一步了：

```
T-001  →  收到训练请求（还没开始）
T-002  →  输出目录已创建
T-003  →  训练进程已启动
T-104  →  epoch 4 进度更新
T-104E →  崩了，在 epoch 4 处
```

我知道它死在 T-003 和 T-104 之间，立刻定位到首轮训练代码段。

### 1.4 状态码生命周期

每个状态码都有生命周期。随着代码重构和功能演进，状态码可能被废弃或删除。生命周期标记告诉 AI：这个码是否仍然可信。

| 标记 | 含义 | AI 行为 |
|---|---|---|
| 🟢 活跃 | 当前版本正常使用的状态码 | 按文档定义处理 |
| 🟡 废弃 | 曾经使用，当前版本不再产生此码（保留以兼容旧日志/旧截图） | 仍可识别，但应提示"此状态码已废弃，请升级到新版" |
| 🔴 已删除 | 已从代码中完全移除，不应出现在任何场景中 | 如果收到此码，说明使用的是非常旧的版本或数据损坏 |

**生命周期变更流程：**

1. 决定废弃某个状态码 → 在本文档中将其标记改为 🟡，注明废弃版本号和替代码
2. 保留 🟡 至少一个主要版本（让旧日志和旧截图仍可被诊断）
3. 下一个主要版本中改为 🔴（正式删除）
4. 🔴 标记保留一个版本后可从文档中移除

> **当前项目状态：** 项目处于开发初期（V2.0），全部状态码均为 🟢 活跃。随着后续版本迭代，部分状态码可能进入废弃生命周期。

### 2.4 训练 `T` — Python `train.py`

训练是状态码最多的模块，按时间线分为四个阶段。

#### 阶段一：启动前检查（T-001 ~ T-019）

| 状态码 | 类型 | 含义 | 触发时机 |
|---|---|---|---|
| T-001 | 信息 | 收到训练请求 | Rust 传入配置参数 |
| T-002 | 信息 | 输出目录已创建 | 目录就绪 |
| T-003 | 信息 | 数据集路径验证通过 | 路径仍可访问 |
| T-004 | 信息 | 模型路径验证通过 | 文件仍可访问 |
| T-005 | 信息 | 磁盘空间检查通过 | 剩余空间充足 |
| T-006 | 信息 | 设备检查通过 | GPU/CPU 可用 |
| T-007 | 信息 | PyTorch + Ultralytics 版本确认 | 版本号记录到日志 |
| T-008 | 信息 | 训练任务已成功启动 | 所有启动前检查通过 |
| T-004W | 警告 | 磁盘可用空间偏低 | 低于 5GB 但仍可训练 |
| T-003E | 错误 | 数据集路径失效 | 文件被移动或删除 |
| T-004E | 错误 | 模型文件不可访问 | 文件被移动或删除 |
| T-005E | 错误 | 磁盘空间不足 | 低于 1GB，拒绝启动 |
| T-006E | 错误 | 指定设备不可用 | 用户选了 GPU 但无 GPU |

#### 阶段二：训练进行中（T-101 ~ T-199）

| 状态码 | 类型 | 含义 | 触发时机 |
|---|---|---|---|
| T-101 | 进度 | 首个 epoch 开始 | epoch 1 开始 |
| T-102 | 进度 | 首个 epoch 完成 | epoch 1 结束，含 loss/mAP |
| T-103 | 进度 | checkpoint 已保存 | best.pt / last.pt 写入磁盘 |
| T-104 | 进度 | epoch 进度更新 | 每个 epoch 结束触发（携带 epoch/N、loss、mAP50、mAP50-95、lr） |
| T-105 | 进度 | 训练进度 25% | 里程碑，触发 Toast |
| T-106 | 进度 | 训练进度 50% | 里程碑，触发 Toast |
| T-107 | 进度 | 训练进度 75% | 里程碑，触发 Toast |
| T-108 | 进度 | 早停监控触发 | patience 窗口内无提升 |
| T-104W | 警告 | 单 epoch 耗时异常增长 | 比前一 epoch > 3x，可能内存泄漏 |
| T-103W | 警告 | checkpoint 保存较慢 | 单次保存 > 30 秒 |
| T-104E | 错误 | GPU 显存溢出 | CUDA OOM |
| T-105E | 错误 | 训练进程意外崩溃 | 非正常退出，exit code ≠ 0 |
| T-106E | 错误 | 磁盘写入失败 | 输出目录不可写 |

#### 阶段三：暂停与恢复（T-201 ~ T-299）

| 状态码 | 类型 | 含义 | 触发时机 |
|---|---|---|---|
| T-201 | 信息 | 收到暂停信号 | stdin 收到 `pause` |
| T-202 | 信息 | 当前 epoch 完成后暂停 | 优雅暂停，不截断当前轮 |
| T-203 | 信息 | 训练已暂停 | 进程挂起，等待恢复/停止 |
| T-204 | 信息 | 收到恢复信号 | stdin 收到 `resume` |
| T-205 | 信息 | 训练已恢复 | 从断点继续 |
| T-206 | 信息 | 收到停止信号 | stdin 收到 `stop` |
| T-207 | 信息 | 训练已停止 | 优雅退出，保留已产出文件 |
| T-201E | 错误 | 暂停信号响应超时 | 当前 epoch > 5 分钟未响应 |

## 四、你反馈时怎么用

### 4.1 最小反馈

```
T-104E
```

这就够了。我看到就知道：训练跑到某个 epoch 时 GPU OOM。

### 4.2 标准反馈

```
T-001 → T-008 正常 → T-104 到 epoch 4 → T-104E
```

把链条写清楚，我连"是不是启动就崩"都不用判断。

### 4.3 完整反馈（用模板）

```
【类型】功能异常
【页面】新建训练
【严重度】🔴
【状态码链】T-001 → T-008 → T-104(epoch4) → T-104E
【复现率】每次
【补注】批次大小 64，图片 640
```

我拿到这个反馈的完整判断链：
1. T-001 到 T-008 正常 → 不是启动配置问题
2. T-104 到了 epoch 4 → 不是立即崩溃，是负载上去后爆了
3. 批次 64 + 图片 640 → 跟你 GPU 16GB 显存不匹配，改 32 试试

## 五、状态码在哪里产生

| 位置 | 实现方式 |
|---|---|
| Python 引擎 stdout | 每条 JSON 必含 `code` 字段：`{"code": "T-104", "type": "progress", ...}` |
| Rust 后端 | 事件 payload 中携带 `code` 字段 |
| 前端 | JS 内用常量映射 `const CODES = { ... }`，统一从 `api.js` 导入 |

## 六、新增状态码流程

如果后续增加新功能，新增状态码遵循以下规则：

1. 确定模块前缀（已有 8 个模块外可新增）
2. 按执行顺序分配序号，跳过已用号段
3. 错误码加 `E` 后缀，警告加 `W` 后缀
4. 同步更新本文档 + `codes.py`（Python 侧） + `codes.js`（前端侧）
