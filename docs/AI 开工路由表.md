# AI 开工路由表

> 路由表是 AI 开发者的唯一入口。每次开工时**只读这一份**，然后按需跳转到其他文档的指定章节。不要一次性加载全部 7 份文档。

---

## 一、路由表是什么

路由表是 `docs/` 目录下全部项目文档的**导航索引**。它不替代任何文档的内容，只告诉你：面对某个具体任务时，你应该按什么顺序去查阅哪几份文档的哪几个章节。

**设计目标：** 让 AI 在 1 分钟内确定"我现在该读什么"，不需要遍历全部文档。

## 二、首次开工注意事项

### 2.1 如果你是第一次克隆这个项目

> **AI 执行说明**：本小节列出的是人类开发者的环境准备工作。AI 在阅读本小节时，只需要**检查并确认**当前环境是否符合以下要求：如果不符合，**向用户报告差异**，不尝试自行安装或修复。

| 检查项 | 要求 | 验证命令 |
|---|---|---|
| Rust | 1.97+ (MSVC, `stable-x86_64-pc-windows-msvc`) | `rustup show active-toolchain` |
| Node.js | v24+ + npm | `node --version` |
| Python | 3.13 | `python --version` |
| 虚拟环境 | 项目根目录下 `venv\`，依赖 `torch==2.6.0+cu124`、`ultralytics==8.4.71` | 在项目根执行 `.\venv\Scripts\activate` |
| Tauri CLI | 2.x | `cargo tauri --version` |
| VS Build Tools | 2026（C++ 桌面开发工作负荷） | `where cl.exe`（非空即可） |
| WebView2 | Windows 10 1803+ 已内置 | 无需手动安装 |
| NSIS | ≥ 3.0（打包阶段才需要） | `makensis --version` |

**人类开发者初始化命令（供参考）：**

```powershell
cd <项目根目录>   # 即包含 docs\ 的目录（git 仓库根）
python -m venv venv
.\venv\Scripts\activate
pip install torch torchvision torchaudio --index-url https://download.pytorch.org/whl/cu124
pip install ultralytics opencv-python matplotlib pillow pyyaml numpy
python -c "import torch; print(torch.cuda.is_available())"  # 必须输出 True
```

> Rust、Node、Tauri CLI、VS Build Tools、WebView2 这五项属于二进制工具检查，通过命令行直接验证（`rustup show` / `node --version` / `cargo tauri --version` / `where cl.exe`），无需对应的 Python 状态码。仅 Python/PyTorch/CUDA/GPU/磁盘空间需要通过 `env_check.py` 返回状态码（E-001~E-007）。

### 2.2 关于虚拟环境

- 虚拟环境目录：**项目根目录下的 `venv\`**——相对约定，不写死盘符，项目整体移动后约定依然成立
- **AI 定位规则**：以当前项目根（含 `docs\` 的目录 / git 仓库根）拼出 `venv\` 路径。找不到 → 向人类报告并请其按 2.1 命令创建，**不自行创建**
- **每次开发前**先激活：在项目根执行 `.\venv\Scripts\activate`
- 虚拟环境已加入 `.gitignore`，不提交到仓库
- 如果虚拟环境损坏：删除 `venv` 目录，按 2.1 的初始化命令重建
- 打包阶段 Python 依赖会单独嵌入安装包，与开发用虚拟环境无关

### 2.3 如果你是恢复一个中断的项目

1. **首先读 SESSION.md**（`docs/SESSION.md`）——这里记录了上次做到哪、下一步是什么、已知问题。10 秒恢复上下文
2. **⚠️ SESSION 过时检测：** 对比 SESSION.md 最后更新时间与 `git log --oneline -1 --format=%ci` 的最新 commit 时间。如果 SESSION 时间早于最新 commit **且差异 > 1 天** → SESSION 可能过时。先运行 `git log --oneline -5` 了解最新变更，再读 SESSION（将其视为"可能不完整的参考"而非"绝对事实"）
3. 如果 SESSION.md 不存在或确认过时：读路由表，找到你上次中断的任务对应行
4. 按 ① → ② → ③ 顺序翻阅即可重拾上下文
5. 如果中断超过一周，建议额外阅读"开始写代码前的总览"行

> **AI 执行规则**：每次完成一个任务后，必须更新 SESSION.md（当前阶段、上次做到、下一步、已知问题）。SESSION 是 AI 之间交接上下文的唯一通道——**上一个 AI 忘了更新 = 你拿到的是假信息**。启动时的 `git log` 对比是防止被假信息误导的最后防线。

## 三、如何使用路由表

### 3.1 基本用法

```
1. 在下表中找到你当前的任务场景
2. 按 ① → ② → ③ 的顺序依次查阅对应文档
3. 查阅完成后开始工作，不要加载其余文档
```

> **自导航规则**：如果任务场景不在路由表中，按以下优先级自行查找：**需求（做什么） > 计划（怎么做） > 设计风格 / 状态码（约束） > UI 模型 / 报告模型（长什么样）**。这个优先级 = 功能正确性永远优先于视觉一致性。

### 3.2 查阅规则

- **①** 是你最需要的文档，提供了该任务的核心约束和规范
- **②③** 是补充文档，确保你不会遗漏影响其他模块的约束
- 如果 ① 已经足够完成任务，②③ 可以略读（扫一眼确认无冲突即可）
- 如果你发现某行缺少了你实际需要的文档，记录并建议更新本表

### 3.3 文档缩写速查

| 缩写 | 全称 |
|---|---|
| 路由表 | AI 开工路由表（路由表） |
| SESSION | 当前会话状态（SESSION.md） |
| ADR | 架构决策记录（`docs/架构决策记录.md`） |
| 需求 | YOLO 模型训练工具 — 需求文档 2.0 |
| 计划 | YOLO 模型训练工具 — 开发计划书 |
| UI 模型 | YOLO 模型训练工具 — 前端 UI 模型图 2.0 |
| 设计风格 | YOLO 模型训练工具 — 设计风格指南 |
| 报告模型 | YOLO 模型训练工具 — 训练报告 HTML 模型图 |
| 测试计划 | YOLO 模型训练工具 — 测试计划书 |
| 状态码 | YOLO 模型训练工具 — 状态码协议文档 |

### 3.4 文档冲突优先级

当两份文档对同一事物的描述不一致时，按以下优先级裁决。交叉约束场景（一份文档引用另一份的约束）以被引用文档的优先级为准。

| 优先级 | 文档 | 裁决范围 |
|---|---|---|
| **1** | 需求文档 | 功能行为、数据格式、错误处理逻辑。（最高优先级，不可被其他文档覆盖） |
| **2** | 状态码协议 | 错误码定义、消息格式。（与需求文档同级，补充关系非替代关系） |
| **3** | 设计风格指南 | 颜色、字体、间距、文案语气、组件交互规范 |
| **4** | UI 模型图 | 布局结构、页面内元素排列 |
| **5** | 报告模型 | 训练报告 HTML 的布局和内容。（仅报告页，不与应用 UI 冲突） |
| **6** | 开发计划书 | 技术栈版本、模块参数格式。（描述性文档，冲突时以本地实际环境为准） |

> 示例：需求 3.1 描述了 "开始训练按钮置灰"，但未指定颜色值 → 以设计风格指南为准。需求 3.1.2 和 UI 模型图对参数展开了位置描述不一致 → 需求文档胜出。

### 3.5 文档变更影响分析

修改某份文档时，必须检查是否影响了其他文档。以下矩阵标明每份文档的"被影响范围"——修改 A 文档时，必须同步检查 B 列中标记的文档是否也需要更新。

| 修改这份文档 ↓ | 需要同步检查的文档 → |
|---|---|
| **需求文档** | 计划（功能变更影响实现方案）、UI 模型（交互变更影响界面）、报告模型（输出项变更影响报告）、状态码（新功能需要新码）、测试计划（新增测试场景） |
| **计划** | 状态码（技术参数变更可能需要新码）、测试计划（架构变更影响测试策略） |
| **设计风格** | UI 模型（样式变更不直接影响布局，但需确认不冲突） |
| **UI 模型** | 计划（布局变更可能影响前端 JS 结构）、测试计划（新页面/状态需要新测试项） |
| **报告模型** | 计划（报告结构变更影响 Python 报告生成模块 `train.py`） |
| **状态码** | （影响面最广的文档之一）Python 引擎代码（`codes.py`）、前端代码（`codes.js`）、测试计划（IPC 面板解读）、路由表（状态码速查引用） |
| **测试计划** | 计划（测试策略变更可能要求增加诊断工具）、状态码（新增诊断码） |
| **ADR** | （ADR 是历史记录，不可修改。如需推翻已有决策，在该文件末尾新增条目说明"为何推翻"，而非直接修改原条目） |

> **使用方式：** 当你被要求修改某份文档时，先查上表中"需要同步检查的文档"列，在修改完成后逐一确认这些文档是否仍然一致。例如：需求文档加了"模型版本对比功能" → 必须检查计划（新增 Python 模块？）、UI 模型（新增页面？）、状态码（新号段？）、测试计划（新测试场景？）。

## 四、AI 开工路由表

### 4.1 项目启动类

| 我在做什么 | 按此顺序查阅文档（① → ② → ③） |
|---|---|
| **首次启动项目，不了解上下文** | ① 路由表 2.1（环境准备：先确认 Rust/Node/Python/虚拟环境/CUDA 全部就绪，再谈开发） → ② SESSION.md（了解当前进度，可能已有前人做了一些工作） → ③ 需求 1.3~1.5（一句话定位、设计理念、版本定位） → ④ 计划 1.1（技术栈确认，知道有哪些层） → ⑤ 状态码（了解通信语言和模块划分） |
| **理解架构决策（"为什么选 Tauri 而不是 Electron？"）** | ① ADR（`docs/架构决策记录.md`，5 个决策约 5 页，一次读完即可理解全部历史选择） → ② 计划 5.3（数据流架构图，对应 ADR-4 日志报告分离的具体实现） → ③ 需求 5.3（关键工程决策概述，与 ADR 形成"概述+详析"对照） |
| **环境就绪后，验证所有模块是否正常工作** | ① 测试计划 5.1（运行 `python tests/training_state_machine.py`，预期全部 PASS） → ② 如失败，按状态码定位具体模块 → ③ 测试计划 5.2（确认后端没问题再开始前端开发，避免前端调前端和后端同时有 bug 互相干扰） |
| **准备开始写代码（冻结基线）** | ① 需求（最后确认理解无歧义） → ② 状态码（确认模块划分无误） → ③ 在 SESSION.md 中记录："基线已冻结。需求 🔒 硬冻结，设计风格/UI 模型 🔐 软冻结。如有偏离，将通过变更请求报告。" **此后，文档优先于代码：任何偏离必须先提变更请求。** |
| **开始写代码前的总览** | ① 计划（了解当前处于哪个阶段） → ② 需求（理解产品目标和用户故事） → ③ 其余 5 份快速略读（知道有这些约束存在即可） |
| **项目中断几天后恢复** | ① SESSION.md（10 秒恢复上下文：当前阶段、上次做到、下一步、已知问题） → ② 路由表（如果 SESSION 过时，找到上次任务的索引行） → ③ 按索引行查阅。如超过一周，补读"首次启动项目"行 |
| **需要做技术决策（选型/架构调整）** | ① ADR（先读已有决策记录，理解历史选择的理由） → ② 计划（确认调整影响的技术层和模块） → ③ 需求（确认不违背产品定位）。如果决策是全新的，按 ADR 模板新增一份决策记录 |
| **实现中发现无法按文档执行（需偏离规范）** | ① 暂停编码 → ② 按下方模板提交 ≤ 10 行的变更请求，含二选一方案 → ③ 等待人类回复，超时后按下方"超时分级策略"执行 → ④ 人类回复后，先更新受影响文档，再改代码 |

> **变更请求模板（AI 在实现中偏离规范时使用）：**
>
> ```
> 【关联规范】需求 3.1.x / 设计风格 x.x / ...
> 【冲突描述】规范要求 XXX，但实现中发现 YYY
> 【建议方案】
>   A) ...（推荐方案）
>   B) ...
> 【影响范围】需同步更新的文档及章节
> 【超时策略】🔴功能 / 🟡视觉 / 🟢技术 — 超时后按推荐方案继续/等待
> ```
>
> 不超过 10 行。必须二选一，不用开放式提问。
>
> **超时分级策略（人类未回复时。分级依据 = 推翻成本：改按钮颜色的成本远低于改核心流程行为）：**
>
> | 变更类型 | 超时 | 超时后 AI 行为 | 事后推翻 |
> |---|---|---|---|
> | 🔴 功能行为变更（影响需求中的行为描述） | 4 小时 | 暂停该任务，切换到不依赖此决策的独立任务，**不自行决定** | 人类回复后继续原任务；如选了不同方案，回滚已做的推测性工作 |
> | 🟡 交互/视觉微调（不影响功能，仅影响体验） | 2 小时 | 按推荐方案（A）继续，在 SESSION.md 记录"未经确认的决策" | 可事后推翻，成本通常 < 1 小时（CSS/文案级） |
> | 🟢 纯技术选择（等效方案，无用户可见差异） | 30 分钟 | 自行选择，SESSION.md 与 commit 标注 `[tech-decision]` | 可事后讨论，通常无需推翻 |
>
> **超时计时规则：** 工作日 9:00–18:00 提交的按上表计时；18:00 后提交的从次日 9:00 起算；周末提交的从周一 9:00 起算（不把睡眠时间算进超时）。

### 4.2 开发实现类

| 我在做什么 | 按此顺序查阅文档（① → ② → ③） |
|---|---|
| **实现某个前端页面或组件** | ① UI 模型图（照抄布局、状态切换、交互流程） → ② 设计风格（套用颜色变量、字体、圆角、文案语气） → ③ 需求对应章节（核对业务逻辑和数据流） |
| **实现 Python 引擎模块（如 train.py / dataset_check.py）** | ① 计划第 4 章（输入参数、输出格式、JSONL 协议） → ② 状态码对应号段（编码规范，如 T-xxx / D-xxx） + 设计风格（如果该模块产出任何前端可见的视觉元素——日志行、标注框、报告图表、导出图片——必须查设计风格确认颜色和字体，不得硬编码） → ③ 需求 3.1（功能边界和校验规则） |
| **实现 Rust 后端（进程管理 / Tauri Commands / Events）** | ① 计划第 5 章（Command 列表、事件名、存储路径） → ② 状态码 R-xxx（Rust 侧错误码） → ③ 需求 5.3（数据流架构图，确认消息方向） |
| **生成训练报告 HTML** | ① 报告模型（逐区域对照，直接复用布局） → ② 需求 3.1.4（输出项定义和文件路径） |
| **修改 UI 样式 / 调整文案 / 统一语言风格** | ① 设计风格（找颜色变量、字号、语气规则） → ② UI 模型图（确认改的是哪个状态的哪个元素，不误改其他状态） |

### 4.3 测试反馈类

| 我在做什么 | 按此顺序查阅文档（① → ② → ③） |
|---|---|
| **收到测试反馈（含状态码链，如 T-001 → T-104 → T-104E）** | ① 状态码（直接解码，定位到具体文件和行号） → ② 测试计划 3.1（IPC 诊断面板截图解读，辅助判断上下文） → ③ 根据码前缀跳转到对应模块代码（D- → dataset_check.py，T- → train.py，R- → Rust process_manager） |
| **收到测试反馈（只有截图，无状态码）** | ① 测试计划 3.1（IPC 面板截图解读，从截图中提取状态码） → ② UI 模型图（对照预期界面状态） → ③ 若无状态码，则按截图中的现象反查需求文档对应章节 |
| **收到测试反馈（"命令行正常但前端无响应" / 跨层链路不通）** | ① 计划 5.3（数据流架构图，确认消息路径：前端 → Rust → Python） → ② 测试计划 3.1（IPC 面板，看消息在哪一层断了） → ③ 状态码（定位具体是哪条消息的状态码缺失或异常） |
| **收到测试反馈（体验类 / 建议类，无 Bug）** | ① 设计风格（核对是否偏离了语言风格或交互原则） → ② 需求（确认是否违背了原始产品定位） → ③ 如合理，直接采纳；如不合理，引用文档原文说明理由 |

### 4.4 问题排查类

| 我在做什么 | 按此顺序查阅文档（① → ② → ③） |
|---|---|
| **排查联调问题（端到端流程跑不通）** | ① 测试计划第 5 章（运行 `tests/training_state_machine.py` 状态机测试，定位是哪个路径失败） → ② 计划 5.3（数据流架构图） → ③ 状态码（看失败点对应的状态码） |
| **需求描述模糊，我不确定要做什么** | ① 需求（确认是否理解错了产品目标） → ② 设计风格（判断是不是语言/交互风格上的模糊） → ③ 如果仍然不确定，**直接提出具体的二选一问题**而非开放式提问 |
| **场景不在路由表中，找不到匹配行** | ① 需求（先明确要做什么） → ② 计划（确定属于架构中哪一层） → ③ 状态码（确认不引入新的编码冲突）。如发现需要新增的查阅路径，记录并建议更新本表 |

> 排查完成后，按测试计划 6.1 的 Bug 反馈模板整理结果，形成闭环记录。

### 4.5 维护变更类

## 五、使用示例

> **通用执行流程**：先在路由表第四节中找到对应行，再按该行的 ① → ② → ③ 顺序查阅文档。以下示例省略"查表"动作，只展示核心推理链。

### 示例 1：收到状态码链

**场景：** 测试者反馈 `T-001 → T-008 → T-104(Epoch 4) → T-104E`

1. ① **状态码** → `T-104E` = GPU 显存溢出 (OOM)，发生在 epoch 进度更新期间
2. ② **测试计划 3.1** → 确认 IPC 面板中该行是否红色高亮
3. ③ 跳转 `train.py` 搜索 `T-104E` → 找到 `try-except torch.cuda.OutOfMemoryError` → 修复

**全过程只读了 2 份文档（状态码 + 测试计划），然后直接进入代码。**

### 示例 2：收到截图反馈

**场景：** 测试者发来截图，没有状态码，显示"开始训练"按钮灰色不可点击，数据集已选择。

1. ① **测试计划 3.1** → 截图中是否有 IPC 面板？如果没有 →
2. ② **UI 模型图** → "开始训练"可点击条件 = 数据集校验通过 + 模型校验通过 + 环境正常
3. ③ 反查**需求 3.1.1** → 校验通过后才允许开始训练。截图中无校验结果，定为"校验未自动触发"

### 示例 3：实现新页面

**场景：** AI 需要实现「推理测试」页面。

1. ① **UI 模型图**第十、十一节 → 三个区域（模型选择 / 图片上传 / 检测结果）+ 参数展开 + 置灰状态
2. ② **设计风格** → 套用 CSS 变量（`--bg-primary`、`--text-primary`、`--color-primary` 等）
3. ③ **需求 3.3** → 导出 JSON 格式、推理参数默认值、从训练页跳转的逻辑

## 六、维护规则

- 路由表是唯一入口，应保持简洁（目标：不超过 3 页）
- 新增文档时，必须在路由表中补充对应的索引行
- **新增或修改文档时，按 3.5 文档变更影响矩阵检查关联文档是否需要同步更新**
- 如果某行的查阅顺序在实践中发现不合理（比如 ② 应该在 ① 之前），直接改
- 每次项目迭代结束后，对照实际使用体验更新本表
- **SESSION.md 由 AI 在每次完成任务后更新**——人类不需要手动维护它。上一个 AI 忘了更新 = 下一个 AI 拿到假信息
- **ADR 决策记录只增不改**——如需推翻已有决策，新增条目说明"为何推翻"
- **Git Commit 必须标注关联文档版本**——使用标签格式：`[需求v2.0]` `[状态码]` `[ADR]` `[UI模型]` `[设计风格]` 等。可通过 `git log --grep="需求v2.0"` 追溯某版本文档对应的全部代码变更
