## 一、文档目的

本文档定义 AI 开发者（Claude）与测试者（你）之间**如何高效协作**。核心目标：

- 你发现问题后，**1 分钟内写完反馈**，不需要追问我
- 我收到反馈后，**不需要追问你细节**，直接定位代码
- 每轮迭代只改一个点，**改完立刻能复测**，不积压

## 三、内置诊断工具

### 3.1 IPC 诊断面板

**位置：** 开发环境按 `Ctrl+Shift+D` 打开，发布版本不包含此功能。

**作用：** 实时展示前端 ↔ Rust ↔ Python 之间的每一条消息。

```
+─ IPC 诊断面板 ────────────────────────────────────────────+
│                                                            │
│ [全部] [Command] [Event] [Python]  筛选                    │
│ ─────────────────────────────────────────────────────────  │
│                                                            │
│ 14:32:01.234 → CMD  check_dataset("C:/data/game_ui")      │
│ 14:32:01.456 ← PY   {"valid":true,"stats":{...}}          │
│ 14:32:01.458 ← EVT  dataset:check:result → Frontend       │
│ 14:32:05.123 → CMD  start_training({...})                 │
│ 14:32:05.345 ← PY   {"type":"started"}                    │
│ 14:32:06.012 ← PY   {"type":"progress","epoch":1,...}     │
│ 14:32:06.014 ← EVT  train:progress → Frontend              │
│                                                            │
│ 总计: 342 条  错误: 0   ↑ 暂停  📋 导出   🗑 清空          │
│                                                            │
+────────────────────────────────────────────────────────────+
```

**排版规则：**

| 符号 | 含义 |
|---|---|
| `→ CMD` | 前端调用 Tauri Command |
| `← PY` | Python 通过 stdout 返回 |
| `← EVT` | Rust 推送 Tauri Event 到前端 |
| `→ PY` | Rust 通过 stdin 发给 Python |
| `!! CMD` | Command 返回了错误 |
| `!! PY` | Python 进程崩溃 |

**你使用场景：**
- 点了「开始训练」没反应 → 打开面板看 CMD 是否发出、PY 是否响应
- 日志停了但进度条还在跑 → 看 PY 行是否还有输出
- 反馈时截图面板最后 5~10 行，我直接定位断点

### 3.2 Python 模块独立测试命令

每个 Python 引擎模块都可以在命令行独立调用，不依赖前端：

```powershell
# 环境检测
python env_check.py

# 数据集校验
python dataset_check.py --dataset-path "C:/data/game_ui"

# 模型校验
python model_check.py --model-path "yolov8n.pt"

# 推理测试
python infer.py --model-path "best.pt" --image-path "test.jpg" --conf 0.25 --iou 0.45

# 训练（小数据集快速验证）
python train.py --dataset-path "C:/data/mini" --model-path "yolov8n.pt" --epochs 5 --device cuda:0
```

**你使用场景：**
- 前端卡住但不确定是不是 Python 问题 → 直接跑对应命令行，排除前端嫌疑
- 反馈时标注"命令行可正常执行，前端无响应"或"命令行也报错"，我立刻知道是哪层的问题

## 四、三段式错误码系统

### 4.1 编码规则

所有错误都有唯一编码，格式：`{层级首字母}-{序号}`

| 层级 | 前缀 | 负责层 | 示例 |
|---|---|---|---|
| **D**ataset | `D-xxx` | Python `dataset_check.py` | 数据集格式问题 |
| **M**odel | `M-xxx` | Python `model_check.py` | 模型文件问题 |
| **T**raining | `T-xxx` | Python `train.py` | 训练运行时问题 |
| **I**nference | `I-xxx` | Python `infer.py` | 推理运行时问题 |
| **E**nvironment | `E-xxx` | Python `env_check.py` | 系统环境问题 |
| **R**untime | `R-xxx` | Rust 后端 | 进程管理、文件系统 |
| **U**I | `U-xxx` | 前端 | 渲染、交互逻辑 |

### 4.2 错误码清单（部分）

| 错误码 | 错误信息 | 用户可见恢复建议 |
|---|---|---|
| D-001 | 未找到 data.yaml 文件 | 你的数据集目录里需要包含 data.yaml 配置文件 |
| D-002 | labels 目录为空 | labels 目录里没有标注文件，请检查数据集结构 |
| D-003 | 发现 N 个标签文件无对应图片 | 这些图片没有对应的标签，请检查文件名是否匹配 |
| D-004 | data.yaml 格式不正确 | yaml 文件内容有问题，请检查格式 |
| M-001 | 不是有效的 YOLO 模型文件 | 该文件无法被 Ultralytics 加载，请确认是 .pt 格式的预训练权重 |
| M-002 | 模型文件过小（< 100KB） | 文件可能损坏或不是模型权重文件 |
| M-003 | 文件扩展名不支持 | 请上传 .pt 格式的文件 |
| T-001 | GPU 显存不足 (Out of Memory) | 试试把"批次大小"从 N 改成 N/2 |
| T-002 | 训练进程意外崩溃 | 已产出的模型文件已保留，可通过训练历史断点续训 |
| T-003 | 磁盘空间不足 | 输出目录剩余空间不足 1GB，请清理磁盘或更换输出目录 |
| T-004 | 数据集无法读取 | 训练开始前数据集路径失效，请检查文件是否被移动或删除 |
| T-005 | 模型加载失败 | 训练进程无法加载指定的模型文件 |
| I-001 | 推理超时（30s） | 推理耗时超过 30 秒，请检查模型或图片是否正常 |
| I-002 | 图片格式不支持 | 请使用 JPG / PNG / BMP 格式 |
| E-001 | Python 运行时不可用 | 请重新安装应用 |
| E-002 | PyTorch 未检测到 CUDA | 将使用 CPU 训练（会慢一些）。如需 GPU 加速，请安装 NVIDIA 驱动 |
| E-003 | 未检测到 NVIDIA 显卡 | 只能使用 CPU 训练，速度会慢一些 |
| R-001 | Python 进程启动失败 | 请检查 Python 环境或重新安装应用 |
| R-002 | 任务历史文件损坏 | 可能是因为程序异常关闭导致的 |
| U-001 | 页面渲染异常 | 请尝试刷新或重启应用 |

### 4.3 错误码在哪显示

- **Toast 通知**：简短提示 + 错误码
- **日志终端**：详细错误信息 + 错误码 + 建议
- **IPC 诊断面板**：错误码高亮显示
- **训练历史**：异常中断的任务附带错误码

### 4.4 你反馈时怎么用

```
U-001 页面白屏，概率复现
```

这就是一个完整的反馈——不需要描述症状、不需要截图。我看到 `U-001` 就知道是前端渲染异常。

## 五、训练状态机自动化测试

### 5.1 测试脚本

`tests/training_state_machine.py` 会枚举所有合法的状态路径，每条路径用小数据集跑 3 轮，验证训练能正常结束。

**覆盖的路径：**

```
路径 1: 空闲 → 训练中 → 完成
路径 2: 空闲 → 训练中 → 暂停 → 恢复 → 完成
路径 3: 空闲 → 训练中 → 停止（已产出 checkpoint）
路径 4: 空闲 → 训练中 → 停止（无 checkpoint）
路径 5: 空闲 → 训练中 → 崩溃 → 断点续训 → 完成
路径 6: 空闲 → 完整训练 → 后续再训练（同数据集）
```

**运行时只需要一条命令：**

```powershell
python tests/training_state_machine.py
```

**输出格式：**

```
✅ 路径 1: 空闲→训练中→完成 .................. PASS (23s)
✅ 路径 2: 空闲→训练中→暂停→恢复→完成 ........ PASS (38s)
❌ 路径 3: 空闲→训练中→停止 .................. FAIL — last.pt 未保存
✅ 路径 4: 空闲→训练中→崩溃→续训→完成 ........ PASS (41s)
───
4/4 通过 | 0 失败 | 耗时 1m42s
```

### 5.2 使用规则

- **开发阶段：** 每次改了 `train.py` 或 Rust 进程管理代码后跑一次
- **集成阶段：** 每次联调前先跑，全绿再开始测前端
- **你测前端前：** 跑一遍确认后端没问题，省得前端卡住时还得排查是不是后端崩了

### 5.3 AI 自动化测试集成

以下规则定义了 AI 开发者如何与自动化测试交互——不需要人类提醒。

#### 5.3.1 AI 提交前自检

**AI 在每次提交代码前，必须执行以下检查：**

```
1. python tests/training_state_machine.py
2. 如果全部 PASS → 可以提交
3. 如果有 FAIL → 先修复，再提交
4. 如果测试脚本本身不可用（如项目初期 tests/ 目录未创建）→ 跳过，但在 SESSION.md 中备注
```

**AI 的 SESSION.md 更新模板（测试结果）：**

```markdown
## 最近测试结果

2026-07-18 14:32 | training_state_machine.py | 6/6 PASS | 耗时 1m42s
```

#### 5.3.2 测试结果文件

测试脚本应每次运行时将结果写入 `tests/results/latest.json`：

```json
{
  "timestamp": "2026-07-18T14:32:00",
  "script": "training_state_machine.py",
  "total": 6,
  "passed": 6,
  "failed": 0,
  "duration_seconds": 102,
  "failures": []
}
```

**AI 在开始新工作前，先检查 `tests/results/latest.json`：**

- 如果上次测试全绿且距今 < 24 小时 → 环境健康，直接开始新任务
- 如果上次测试有失败 → 先修复失败项，再开始新任务（不把 Bug 留给下一次提交）
- 如果文件不存在或距今 > 24 小时 → 运行一次 `training_state_machine.py` 刷新结果

#### 5.3.3 CI 集成（GitHub Actions）

在 GitHub 仓库中配置自动化工作流：

```yaml
# .github/workflows/test.yml
name: State Machine Tests
on: [push, pull_request]
jobs:
  test:
    runs-on: windows-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.13'
      - run: pip install torch ultralytics opencv-python
      - run: python tests/training_state_machine.py
```

**AI 的行为：** 提交代码后，如果 `git push` 触发 CI 失败，AI 应在收到 GitHub 通知后定位失败原因并修复。

#### 5.3.4 测试优先原则

当 AI 面对以下场景时，必须先跑测试再改代码：

| 场景 | AI 行为 |
|---|---|
| 修改 `train.py` | 改完后跑 `training_state_machine.py`，确认 6 条路径全绿 |
| 修改 Rust 进程管理 | 同上（Rust 侧改动直接影响 Python 子进程生命周期） |
| 修改 `dataset_check.py` | 在命令行用合法/非法数据集各测一次 |
| 修改 `infer.py` | 用已知模型 + 已知图片测试推理结果一致性 |
| 联调前 | 跑全部可用的自动化测试，全绿再通知人类开始联调 |
| 收到 Bug 反馈 | 先跑测试确认当前基线状态，再定位和修复 |

> **设计理由：** AI 编程速度快，容易在上一个修改中引入回归。自动化测试 + 提交前强制运行 = 把"AI 自己改坏了"的概率降到最低。

## 六、Bug 反馈模板

### 6.1 模板

```
【类型】功能异常 / 体验问题 / 崩溃 / 建议
【页面】新建训练 / 训练历史 / 推理测试 / 设置 / 全局
【严重度】🔴 阻断（无法继续） / 🟡 功能缺陷 / 🟢 小问题
【操作步骤】
1. xxx
2. xxx
3. xxx
【预期结果】
xxx
【实际结果】
xxx
【诊断码】D-xxx / T-xxx（如果有）
【复现率】每次 / 概率（约 X%）
【附件】截图 / IPC面板截图 / 命令行输出
```

### 6.2 示例

```
【类型】功能异常
【页面】新建训练
【严重度】🟡 功能缺陷
【操作步骤】
1. 选好数据集，校验通过
2. 直接把「批次大小」拖到 256
3. 点开始训练
【预期结果】
正常开始训练
【实际结果】
页面卡住 10 秒后报错，看不到具体错误信息
【诊断码】T-001
【复现率】每次
【附件】[截图]
```

### 6.3 模板比你自由描述好在哪

- 你按格式写，我**不需要追问**"你选的是哪个数据集？什么操作顺序？"
- 每次反馈都是一条独立记录，积压 10 条我按严重度排序改，不会遗漏
- 你也可以用它来提**体验建议**——"这个按钮颜色太暗看不清"跟 Bug 同等重视

## 七、协作流程

### 7.1 开发阶段（前 6 周）

```
我：写代码 → 自测通过 → 提交
你：拉代码 → 对照需求文档测功能 → 按模板反馈
我：按严重度排序修 → 提交 → 循环
```

**节奏：** 我每完成一个模块（如"数据集校验前端"），通知你测。你测完反馈，我当天或次日修完。

### 7.2 联调阶段（第 7~9 周）

```
我：端到端流程自测通过
你：完整流程跑一遍，重点关注跨页面跳转、异常场景
每次反馈前先打开 IPC 诊断面板
```

### 7.3 打包测试阶段（第 10~12 周）

```
我：产出一个安装包
你：在不同机器上安装测试，重点测兼容性
反馈格式同上，额外标注机器配置（Win10/Win11、GPU 型号、内存）
```

## 八、反馈优先级

我收到反馈后按以下顺序处理：

| 优先级 | 类型 | 示例 |
|---|---|---|
| P0 | 🔴 阻断性 Bug | 训练无法开始、应用崩溃、数据丢失 |
| P1 | 🟡 核心功能缺陷 | 暂停无效、断点续训恢复失败、报告不生成 |
| P2 | 🟡 非核心功能缺陷 | 推理页某个按钮无响应、历史搜索不准 |
| P3 | 🟢 体验问题 / 文本错误 | Tooltip 文案不清晰、颜色不对 |
| P4 | 💡 改进建议 | 希望某个交互换一种方式 |

## 九、测试数据集

项目附带两个专用测试数据集，位于 `tests/datasets/`：

| 数据集 | 图片数 | 类别数 | 用途 |
|---|---|---|---|
| `mini_valid` | 10 | 3 | 快速验证：训练 5 轮 < 30 秒 |
| `full_sample` | 50 | 3 | 体验测试：模拟真实训练流程 |

状态机测试用 `mini_valid`，你测前端功能用 `full_sample`。

## 十、测试检查清单

每个阶段测完后逐项打勾：

### 10.1 新建训练

- [ ] 数据集校验通过 → 绿色提示 + 统计信息正确
- [ ] 数据集校验失败 → 红色提示 + 逐条列出具体问题
- [ ] 数据集预览 → 标注框正确叠加，翻页正常
- [ ] 模型校验通过 → 显示模型信息
- [ ] 模型校验失败 → 红色提示 + 下载引导链接
- [ ] 训练 5 轮 → 日志实时更新、进度条正常、指标表格有数据
- [ ] 暂停 → 日志显示"已暂停"，参数区锁定
- [ ] 恢复 → 从断点继续，指标接续正确
- [ ] 停止 → 二次确认弹窗，文件保留
- [ ] 训练完成 → 摘要正确，报告可打开
- [ ] GPU OOM → 友好错误提示 + 建议调小批次
- [ ] 磁盘不足 → 训练前弹出警告
- [ ] 训练最小化/切页面 → 训练继续
- [ ] 训练中关窗口 → 确认弹窗 + 文件保留
- [ ] 断点续训 → 加载配置 + 训练从 epoch 0 开始

### 10.2 训练历史

- [ ] 任务列表正确显示（状态、指标、时间）
- [ ] 搜索/筛选/排序正常
- [ ] 查看报告 → 浏览器打开 HTML
- [ ] 下载模型 → 打开文件夹
- [ ] 断点续训 → 跳转新建训练并预填参数
- [ ] 删除任务 → 二次确认 + 文件删除
- [ ] 批量删除正常
- [ ] 空状态显示引导

### 10.3 推理测试

- [ ] 模型加载 → 显示模型信息
- [ ] 拖拽/点击上传图片
- [ ] 推理完成 → 标注结果图 + 检测表 + 统计正确
- [ ] 导出结果图 + 导出 JSON
- [ ] 置信度/IoU 参数生效
- [ ] 未选模型时图片上传区置灰
- [ ] 训练完成页跳转 → 自动加载模型

### 10.4 设置

- [ ] 默认输出目录可修改
- [ ] 日志级别切换生效
- [ ] 环境检测结果正确（Python/PyTorch/GPU/磁盘）
- [ ] 模型下载指引内容完整
- [ ] 侧边栏环境指示灯正确

### 10.5 全局

- [ ] 侧边栏导航切换流畅
- [ ] Toast 通知正常显示/消失
- [ ] 训练状态灯随状态变化
- [ ] 暗色主题一致性好
